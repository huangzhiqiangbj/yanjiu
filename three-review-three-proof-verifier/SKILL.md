# 三审三校成稿核验

## 目标
- 这个 skill 不负责代写成稿，负责在交付前做最后一道核验门。
- 核验对象不是“文章看起来差不多”，而是“当前任务要求 + 上游 skill 自带检查项”是否已经被真正执行并留下证据。
- 没有证据，不算完成；有阻断项，不得对用户声称“已完成”“可交付”。

## 何时必须触发
- 当前任务已经进入 `准备交付`、`准备导出`、`准备说完成` 的阶段。
- 当前主 skill 含有 `自检清单`、`质量检查`、`交付前自检`、`验证`、`核验`、`验收` 等章节。
- 输出承诺包含 `Markdown`、`Word`、`HTML`、文件落盘、候选池、事实包、评分表、参考来源、时间窗核验等任一项。
- 用户显式要求“检查成稿”“复核”“把关”“三审三校”“验收”。

## 核心纪律
1. 先读任务要求，再读主 skill，再看成稿，最后下结论。
2. 先找证据，再判断是否完成；不能倒过来。
3. 上游 skill 中写了 `已...`、`必须...`、`默认...` 的条目，默认按必检项处理；只有明确写了条件的，才能标为条件项。
4. 任何 `应该`、`看起来`、`大致`、`差不多` 都不能作为通过依据。
5. 发现阻断项时，先列缺口和补动作；不要直接美化成“基本完成”。

## 最低输入
核验前至少掌握这四类对象：
- `当前任务要求`：用户要交付什么、给谁看、是否要求落盘或导出。
- `主 skill`：当前真正驱动成稿的 `SKILL.md`。
- `成稿对象`：对话中的最终文本，或已生成的 `Markdown/Word/HTML` 文件。
- `补充证据`：候选池、事实包、评分记录、文件存在性、已执行的验证结果等。

若任一对象缺失，先说明“无法完成完整核验的原因”，再列出缺失项；不要装作已核验。

## 工作流

### 第一步：抽取原始检查项
从主 skill 中优先抽取以下章节的全部条目：
- `## 自检清单`
- `## 质量检查`
- `## 交付前自检`
- `## 验证`
- `## 核验`
- `## 验收`
- 其他语义等价章节，如 `交付前确认`、`完成前检查`、`发布前检查`

抽取后把条目归并为四类：
- `任务完成项`：板块、标题、输出格式、读者对象、是否先候选后成稿。
- `事实证据项`：日期、主体、时间窗、数据、来源、标题与正文是否一致。
- `表达结构项`：篇幅、语体、标题、小标题、是否重复事实、是否解释机制或边界。
- `交付落盘项`：Markdown/Word/HTML 是否生成，文件夹是否正确，是否存在承诺产物。

### 第二步：执行“三审”
#### 一审：任务契约审
逐项核对：
- 成稿是否覆盖了任务要求和主 skill 承诺的输出结构。
- 是否漏掉必须出现的章节、字段、读者适配或中间步骤。
- 是否把“先候选后成稿”“先核验后交付”这类流程性要求真正执行了。

#### 二审：事实证据审
逐项核对：
- 关键日期、机构、标题、对象、范围、数字、时间窗是否有依据。
- “已核验”“已确认”“已落盘”“已调用某 sibling skill”是否有可见证据支撑。
- 标题结论是否能被正文和来源支持；若不能，记为阻断项。

#### 三审：表达交付审
逐项核对：
- 结构是否符合主 skill 的句式、篇幅、标题和语体要求。
- 是否把 `事件` 事实在分析段里机械复述。
- 是否遗漏机制、边界、条件、迁移前提、治理约束等主 skill 明确要求的判断层内容。
- 是否存在模板化空话、越位结论、与目标读者不匹配的表述。

### 第三步：执行“三校”
#### 一校：逐条对照校
对原始检查项逐条打标：
- `通过`：找到了直接证据
- `部分通过`：只满足一部分，或仍缺边界条件
- `不通过`：没有证据，或与成稿相矛盾
- `不适用`：主 skill 明确写了条件，且本任务不触发

每条至少写出一句证据说明。没有证据说明的“通过”无效。

#### 二校：矛盾遗漏校
重点查四类问题：
- 主 skill 里要求了，但成稿里完全没出现
- 成稿里声称做了，但看不到证据
- 文件或导出承诺存在，但产物不存在或不完整
- 多处内容彼此矛盾，例如日期、标题、范围、章节名不一致

#### 三校：交付门禁校
按以下标准给结论：
- `通过`：所有必检项有证据，且没有阻断项
- `有条件通过`：无阻断项，但仍有少量非关键缺口，必须向用户明说
- `不通过`：任一阻断项未解决，或关键项无法给出证据

## 什么算阻断项
出现任一情况，默认 `不通过`：
- 主 skill 明确要求的核心章节、核心字段或核心输出缺失
- 时间窗、来源、主体、日期等关键事实未核验却被写成已确认
- 承诺导出 `Markdown`、`Word`、`HTML`，但缺文件或文件不对应
- 主 skill 明确要求先候选后成稿，但实际跳过候选阶段
- 标题、摘要或主判断强于证据，正文无法支持
- 把未验证推断写成确定性结论
- 用“已完成”“已核验”表述，但拿不出任何证据

## 证据标准
- `已写出`：成稿中能定位到对应内容
- `已核验`：能看到来源、事实包、时间窗、日期或主体对照，或明确核验记录
- `已导出`：能看到文件路径、文件存在或生成结果
- `已调用其他 skill`：能看到该 skill 要求的结果字段或后续产物，而不是只说“已调用”
- `已通过质量检查`：必须能回到原始检查项逐条说明，不接受笼统一句“已检查”

## 输出格式
默认输出一个简洁的核验结论，不要把大段幕后思考直接倒给用户。使用以下结构：

```markdown
核验结论：通过 / 有条件通过 / 不通过

一审：任务契约
- ...

二审：事实证据
- ...

三审：表达交付
- ...

未通过或待补项
- ...

建议补动作
- ...
```

若用户明确要求看细表，再补充：

```markdown
| 原检查项 | 结论 | 证据 | 缺口 |
|---|---|---|---|
| ... | ... | ... | ... |
```

## 交付话术规则
- 只有在 `通过` 或 `有条件通过` 时，才能对用户说“成稿已完成”。
- 若是 `有条件通过`，必须同时说清楚保留缺口。
- 若是 `不通过`，只能说“已完成核验，当前仍有以下阻断项”，不能说“已经基本可以交付”。

## 常见失败方式
- 只读成稿，不回看主 skill 的检查项
- 只看是否“写得像样”，不看是否真的完成核验动作
- 发现 Word、HTML 或 Markdown 缺失，仍然说“已导出”
- 把“需要用户确认后再成稿”的流程要求漏掉
- 把 `对教育的影响`、`思考与启发` 等分析段当成任意发挥，没有检查是否符合上游 skill 的句式和边界要求

## 快速模板
当你被别的 skill 调用时，先在心里跑完这四句：
1. 当前任务到底承诺交付了什么？
2. 上游 skill 明确要求我检查什么？
3. 我手里有什么直接证据？
4. 现在能不能负责任地说“可交付”？

只要第 2 句或第 3 句答不上来，就不能放行。
